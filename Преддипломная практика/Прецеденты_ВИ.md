В таблицах 2–6 приведены спецификации ключевых вариантов использования разрабатываемой системы.

Таблица 2 — Спецификация варианта использования «Регистрация»

+-----------------------------------------------------------------------+
| Прецедент: регистрация                                                |
+=======================================================================+
| ID: UC1                                                               |
+-----------------------------------------------------------------------+
| Краткое описание: клиент создаёт учётную запись в системе,            |
| вводя адрес электронной почты, пароль и полное имя.                   |
+-----------------------------------------------------------------------+
| Действующие лица: клиент                                              |
+-----------------------------------------------------------------------+
| Предусловие: клиент открыл страницу регистрации                       |
+-----------------------------------------------------------------------+
| Основной поток:                                                       |
|                                                                       |
| 1. Клиент вводит адрес электронной почты, пароль и полное имя.        |
|                                                                       |
| 2. Система проверяет корректность формата email и наличие             |
|    обязательных полей.                                                |
|                                                                       |
| 3. Система передаёт данные в сервис аутентификации,                   |
|    который сохраняет учётную запись (пароль хранится                  |
|    в виде bcrypt-хеша).                                               |
|                                                                       |
| 4. Система уведомляет клиента об успешной регистрации                 |
|    и предлагает перейти ко входу в систему.                           |
+-----------------------------------------------------------------------+
| Постусловие: учётная запись клиента создана; клиент может             |
| выполнить вход в систему (UC2).                                       |
+-----------------------------------------------------------------------+
| Альтернативный поток: если адрес электронной почты уже                |
| зарегистрирован, система возвращает сообщение об ошибке.              |
| Если обязательные поля не заполнены — отображается                    |
| предупреждение.                                                       |
+-----------------------------------------------------------------------+

Таблица 3 — Спецификация варианта использования «Вход в систему»

+-----------------------------------------------------------------------+
| Прецедент: вход в систему                                             |
+=======================================================================+
| ID: UC2                                                               |
+-----------------------------------------------------------------------+
| Краткое описание: клиент вводит учётные данные и получает             |
| JWT-токен для доступа к защищённым функциям.                          |
+-----------------------------------------------------------------------+
| Действующие лица: клиент                                              |
+-----------------------------------------------------------------------+
| Предусловие: учётная запись клиента создана (UC1)                     |
+-----------------------------------------------------------------------+
| Основной поток:                                                       |
|                                                                       |
| 1. Клиент вводит адрес электронной почты и пароль.                    |
|                                                                       |
| 2. Система проверяет учётные данные через сервис аутентификации.      |
|                                                                       |
| 3. Сервис аутентификации проверяет пароль и генерирует                |
|    JWT-токен с идентификатором пользователя и ролью.                  |
|                                                                       |
| 4. Клиент получает токен доступа и переходит в личный кабинет.        |
+-----------------------------------------------------------------------+
| Постусловие: клиент аутентифицирован; токен доступа сохранён          |
| и используется при последующих операциях.                             |
+-----------------------------------------------------------------------+
| Альтернативный поток: если учётные данные неверны, система            |
| возвращает сообщение об ошибке аутентификации.                        |
+-----------------------------------------------------------------------+

Таблица 4 — Спецификация варианта использования «Поиск и фильтрация санаториев»

+-----------------------------------------------------------------------+
| Прецедент: поиск и фильтрация санаториев                              |
+=======================================================================+
| ID: UC5 (расширяет UC4 «Просмотр каталога санаториев»)               |
+-----------------------------------------------------------------------+
| Краткое описание: клиент задаёт параметры фильтрации и получает       |
| список санаториев, соответствующих заданным критериям.                |
+-----------------------------------------------------------------------+
| Действующие лица: клиент                                              |
+-----------------------------------------------------------------------+
| Предусловие: клиент открыл страницу каталога санаториев               |
+-----------------------------------------------------------------------+
| Основной поток:                                                       |
|                                                                       |
| 1. Клиент задаёт один или несколько параметров фильтрации:            |
|    город, медицинский профиль, расстояние до моря,                    |
|    ценовой диапазон, даты проживания, количество гостей.              |
|                                                                       |
| 2. Система отправляет запрос с параметрами фильтрации                 |
|    на сервер.                                                         |
|                                                                       |
| 3. Сервер возвращает список санаториев, соответствующих               |
|    критериям, с пагинацией.                                           |
|                                                                       |
| 4. Клиент просматривает результаты и при необходимости                |
|    уточняет параметры подбора.                                        |
+-----------------------------------------------------------------------+
| Постусловие: клиент получил список подходящих санаториев и            |
| может перейти к просмотру деталей (UC6).                              |
+-----------------------------------------------------------------------+
| Альтернативный поток: если ни один санаторий не соответствует         |
| заданным параметрам, система отображает пустой список                 |
| с предложением изменить критерии фильтрации.                          |
+-----------------------------------------------------------------------+

Таблица 5 — Спецификация варианта использования «Создание бронирования»

+-----------------------------------------------------------------------+
| Прецедент: создание бронирования                                      |
+=======================================================================+
| ID: UC7                                                               |
+-----------------------------------------------------------------------+
| Краткое описание: клиент выбирает санаторий, указывает даты           |
| и создаёт бронирование. Включает проверку доступности мест            |
| (UC8), расчёт стоимости (UC9) и оплату (UC10).                        |
+-----------------------------------------------------------------------+
| Действующие лица: клиент                                              |
+-----------------------------------------------------------------------+
| Предусловие: клиент аутентифицирован (UC2); санаторий выбран          |
| в каталоге                                                            |
+-----------------------------------------------------------------------+
| Основной поток:                                                       |
|                                                                       |
| 1. Клиент указывает дату заезда, дату выезда и количество гостей.     |
|                                                                       |
| 2. Система проверяет корректность дат и наличие свободных мест        |
|    на указанный период (UC8).                                         |
|                                                                       |
| 3. Система рассчитывает стоимость бронирования исходя из              |
|    цены санатория и длительности проживания (UC9).                    |
|                                                                       |
| 4. Клиент подтверждает бронирование и переходит к оплате (UC10).      |
|                                                                       |
| 5. Система создаёт бронирование в статусе confirmed и                 |
|    публикует событие в брокер сообщений.                              |
|                                                                       |
| 6. Система возвращает клиенту подтверждение с деталями бронирования.  |
+-----------------------------------------------------------------------+
| Постусловие: бронирование создано и отображается в списке             |
| бронирований клиента (UC11).                                          |
+-----------------------------------------------------------------------+
| Альтернативный поток: если свободных мест нет, система                |
| уведомляет клиента и предлагает выбрать другие даты.                  |
| При некорректных датах (дата выезда ≤ дате заезда) —                  |
| отображается сообщение об ошибке.                                     |
+-----------------------------------------------------------------------+

Таблица 6 — Спецификация варианта использования «Отмена бронирования»

+-----------------------------------------------------------------------+
| Прецедент: отмена бронирования                                        |
+=======================================================================+
| ID: UC13                                                              |
+-----------------------------------------------------------------------+
| Краткое описание: клиент отменяет ранее созданное бронирование.       |
| Система переводит бронирование в статус cancelled.                    |
+-----------------------------------------------------------------------+
| Действующие лица: клиент                                              |
+-----------------------------------------------------------------------+
| Предусловие: клиент аутентифицирован (UC2); бронирование              |
| существует и принадлежит клиенту                                      |
+-----------------------------------------------------------------------+
| Основной поток:                                                       |
|                                                                       |
| 1. Клиент открывает список своих бронирований (UC11).                 |
|                                                                       |
| 2. Клиент выбирает бронирование и инициирует его отмену.              |
|                                                                       |
| 3. Система запрашивает подтверждение действия.                        |
|                                                                       |
| 4. Клиент подтверждает отмену.                                        |
|                                                                       |
| 5. Система переводит бронирование в статус cancelled и                |
|    публикует событие booking.cancelled в брокер сообщений.            |
|                                                                       |
| 6. Система обновляет список бронирований клиента.                     |
+-----------------------------------------------------------------------+
| Постусловие: бронирование отменено; место в санатории на              |
| указанный период освобождается.                                       |
+-----------------------------------------------------------------------+
| Альтернативный поток: если бронирование уже отменено или не           |
| принадлежит данному клиенту, система возвращает ошибку.               |
+-----------------------------------------------------------------------+
