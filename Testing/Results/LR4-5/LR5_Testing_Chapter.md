# Глава 5. Тестирование программного обеспечения

В рамках верификации разработанного микросервиса `deal-service` было проведено три вида тестирования: функциональное тестирование методом чёрного ящика, нагрузочное тестирование и статический анализ метрик кода. Объектом тестирования является сервисный слой модуля бронирования (`BookingService`), реализованный на языке Go.

---

## 5.1 Функциональное тестирование

Функциональное тестирование проводилось методом чёрного ящика с применением техники функциональных диаграмм (cause-effect graph) и таблиц решений. Объектом тестирования является сценарий `CreateBooking` — создание бронирования санатория. Данный метод позволяет систематически охватить все комбинации входных условий и установить однозначное соответствие между ними и ожидаемыми результатами.

### 5.1.1 Причины и следствия

В рамках анализа сценария были выделены 7 входных условий (причин) и 5 возможных результатов (следствий).

Таблица 5.1 — Причины (входные условия)

| ID | Формулировка |
|----|-------------|
| П1 | `client_id` заполнен (не пустая строка) |
| П2 | `sanatorium_id` заполнен (не пустая строка) |
| П3 | `check_in` задана (дата въезда указана) |
| П4 | `check_out` задана (дата выезда указана) |
| П5 | `check_out > check_in` (корректный диапазон дат) |
| П6 | `guests > 0` (количество гостей положительное) |
| П7 | Репозиторная проверка пройдена (санаторий существует, есть свободные места, даты доступны) |

Таблица 5.2 — Следствия (выходные результаты)

| ID | Формулировка |
|----|-------------|
| С1 | Ошибка `client_id and sanatorium_id are required` |
| С2 | Ошибка `guests must be greater than zero` |
| С3 | Ошибка `invalid booking date range` |
| С4 | Доменная ошибка репозитория (`sanatorium not found` / `guests exceed capacity` / `not available`) |
| С5 | Успешное создание бронирования (`status = confirmed`) |

Между причинами действует ограничение Е1: если П3 = 0 или П4 = 0 (хотя бы одна из дат не задана), то П5 не вычисляется. П5 имеет смысл только при П3 = 1 и П4 = 1 одновременно.

Порядок проверок в коде метода `BookingService.CreateBooking`:

## 5.2 Нагрузочное тестирование

Для оценки поведения разработанного приложения под нагрузкой было проведено нагрузочное тестирование с использованием инструмента k6. Целью тестирования являлась проверка корректности работы системы при выполнении типового пользовательского сценария, а также оценка времени отклика основных HTTP-запросов.

В качестве тестируемого сценария был выбран пользовательский поток, включающий регистрацию пользователя, вход в систему, получение списка санаториев и выполнение операции создания бронирования. Таким образом, в рамках одного тестового прохода последовательно выполнялись основные действия, характерные для практического использования приложения.

Нагрузка в ходе тестирования изменялась поэтапно. На первом этапе в течение 30 секунд количество виртуальных пользователей увеличивалось до 10. Далее в течение 1 минуты нагрузка возрастала до 50 виртуальных пользователей, после чего в течение ещё 1 минуты поддерживалась на этом уровне. На завершающем этапе в течение 30 секунд нагрузка снижалась до 0.

В ходе тестирования были заданы пороговые значения для ключевых запросов. Для запроса получения списка санаториев было установлено требование, согласно которому 99% запросов должны выполняться менее чем за 1 секунду. Для запроса создания бронирования было установлено требование, согласно которому 99% запросов должны выполняться менее чем за 2 секунды. Кроме того, для общего числа ошибочных HTTP-запросов было задано ограничение не более 5%.

По результатам тестирования установлено, что проверки корректности регистрации, аутентификации и получения списка санаториев были пройдены в полном объёме: по каждому из этих шагов зафиксировано 3 488 успешных выполнений из 3 488. Операция создания бронирования завершилась с частичными ошибками: из 3 488 попыток успешно выполнено 2 316, что составляет 66,4%. В совокупности из 20 928 проверок успешно пройдено 19 756 (94,40%), доля неуспешных проверок составила 5,60%.

Среднее время выполнения HTTP-запросов по всем операциям составило 358,71 мс, медианное значение — 129,59 мс, 90-й процентиль — 1001,81 мс, 95-й процентиль — 1109,77 мс. Для запроса получения списка санаториев среднее время отклика составило 25,44 мс, значение p(99) — 93,01 мс. Для запроса создания бронирования среднее время отклика составило 40,99 мс, значение p(99) — 135,28 мс.

Заданные пороговые значения по времени отклика выполнены со значительным запасом: для запроса получения санаториев p(99) = 93 мс при установленном ограничении менее 1000 мс, для запроса создания бронирования p(99) = 135 мс при установленном ограничении менее 2000 мс. Вместе с тем пороговое значение по доле ошибочных HTTP-запросов не было достигнуто: показатель `http_req_failed` составил 8,40% при допустимом значении не более 5%.

Источником всех ошибок является операция создания бронирования. Анализ показывает, что отказы обусловлены не деградацией производительности, а конфликтами бизнес-правил при параллельной нагрузке: санаторий с ограниченной вместимостью уже занят на запрашиваемые даты, и сервис корректно отклоняет повторные попытки бронирования. Это поведение является штатным с точки зрения логики приложения.

Таким образом, результаты нагрузочного тестирования показали, что приложение обеспечивает высокую скорость отклика и успешно обрабатывает сценарии регистрации, аутентификации и просмотра данных без ошибок. Превышение порога по доле ошибок обусловлено особенностями тестового сценария и может быть устранено путём расширения тестовых данных либо разделения временны́х диапазонов бронирования между виртуальными пользователями.

## 5.4 Выводы

В ходе тестирования микросервиса `deal-service` были получены следующие результаты.

Функциональное тестирование методом классов эквивалентности для поля `guests` выявило 6 классов (1 правильный и 5 неправильных), все классы покрыты тест-кейсами. Поведение сервиса при граничных и недопустимых значениях соответствует заданным требованиям.

Статический анализ метода `CreateBooking` показал умеренную сложность: объём по Холстеду V = 406,40, цикломатическая сложность CC = 6, метрика Чепина Q = 22. Показатели соответствуют уровню, приемлемому для сопровождения и тестирования.
