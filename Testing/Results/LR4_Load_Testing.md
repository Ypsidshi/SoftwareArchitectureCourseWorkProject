# Нагрузочное тестирование (k6)

Для оценки поведения разработанного приложения под нагрузкой было проведено нагрузочное тестирование с использованием инструмента `k6`. Целью тестирования являлась проверка корректности работы системы при выполнении типового пользовательского сценария, а также оценка времени отклика основных HTTP-запросов.

В качестве тестируемого сценария был выбран пользовательский поток, включающий регистрацию пользователя, вход в систему, получение списка санаториев и выполнение операции создания бронирования. В рамках одного прохода выполнялись действия:
1. `POST /api/v1/users/register` (`auth-service`, порт `8081`).
2. `POST /api/auth/login` (`deal-service`, порт `8082`).
3. `GET /api/sanatoriums`.
4. `POST /api/bookings`.

Нагрузка изменялась поэтапно:
1. `30s` — рост до `10` VU.
2. `1m` — рост до `50` VU.
3. `1m` — удержание `50` VU.
4. `30s` — снижение до `0` VU.

Пороговые значения:
1. `p(99) < 1s` для `GET /api/sanatoriums`.
2. `p(99) < 2s` для `POST /api/bookings`.
3. `http_req_failed <= 5%`.

## Интерпретация

По времени отклика целевые пороги для `list_sanatoriums` и `create_booking` выполнены с запасом (`p(99)` значительно ниже заданных лимитов).

При этом общий критерий по доле ошибок `http_req_failed <= 5%` не выполнен (`8.40%`), поэтому итоговое нагрузочное тестирование считается частично неуспешным.

Основной источник ошибок — операция создания бронирования (`1172` неуспешных проверок на шаге `create_booking`). Это указывает на функционально-валидационные ограничения под параллельной нагрузкой (например, конфликт бизнес-правил бронирования), а не на проблему времени отклика.

Таким образом, система демонстрирует высокую скорость ответа, однако требует доработки стабильности сценария массового создания бронирований при пиковом числе пользователей.
