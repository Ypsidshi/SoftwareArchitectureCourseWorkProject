# ТЗ для AI-агента Codex: выполнение ЛР1 и ЛР2 по тестированию

## 1) Назначение документа
Этот файл — рабочее ТЗ для Codex, чтобы **корректно и полно** выполнить задания из исходного документа `Task on Testing/LR1_2_1_1__1 (1).docx` по:
- ЛР1 «Тестирование черным ящиком»;
- ЛР2 «Тестирование белым ящиком».

Цель: получить структурированный отчет с таблицами, тест-кейсами и пояснениями, пригодный для сдачи.

---

## 2) Что явно требуется в исходном документе

### ЛР1. Тестирование черным ящиком

#### Часть 1. Классы эквивалентности
Нужно:
1. Взять из своей ВКР (или проекта) **одно поле/внутренние данные**.
2. Написать на это поле **техническое задание** (требования и ограничения).
3. Выделить **правильные и неправильные классы эквивалентности**:
   - в табличном виде;
   - в графическом виде.
4. Составить тест-кейсы по КЭ.

Дополнение из методички: предусмотреть тесты для непересекающихся и пересекающихся множеств, если это применимо к структуре поля.

#### Часть 2. Граничные условия
Нужно:
1. По полю из части 1 зафиксировать границы типов:
   - физические,
   - логические,
   - технологические,
   - произвольные (из ТЗ).
2. Если границ одного типа несколько — пронумеровать (например, `Ф1`, `Ф2`, `Т1`...).
3. Составить тесты на границах, в таблице тестов указывать код границы (`Ф1`, `Т2` и т.д.).

#### Часть 3. Функциональная диаграмма и таблица решений
Нужно:
1. Взять **несколько взаимосвязанных полей/переменных** (поведенческий сценарий).
2. Составить список **причин и следствий**.
3. Построить **функциональную диаграмму** (cause-effect graph).
4. Построить **таблицу решений**.
5. Удалить невозможные комбинации и сформировать **тест-кейсы**.

#### Часть 4. Диаграмма состояний и переходов
Нужно:
1. Выбрать **один объект** из проекта (желательно имеющий аналог в БД).
2. Выделить его состояния (взаимоисключающие и покрывающие поведение объекта).
3. Нарисовать S&T-диаграмму (State & Transition), подписать действия на переходах.
4. Пометить на диаграмме **маршруты тестирования через все состояния**.

#### Часть 5. Комбинаторика / попарное тестирование
Нужно:
1. Подготовить входные факторы и значения для инструмента (`pairwise`/`allpairs`).
2. Сформировать файл входных данных для инструмента.
3. Получить результат (скрин или список тест-кейсов).
4. Провести анализ:
   - количество полного комбинаторного перебора;
   - количество тестов при попарном покрытии.

---

### ЛР2. Тестирование белым ящиком
В документе подробно даны критерии и примеры на функции:
```c
void func(int a, int b, float x) {
  if ((a > 1) && (b == 0)) x = x/a;
  if (a == 2 || x > 1) x++;
}
```

Явно описаны части:
1. Покрытие операторов.
2. Покрытие решений.
3. Покрытие условий.
4. Покрытие решений и условий.
5. Комбинаторное покрытие условий.
6. Управляющий граф программы (CFG) и граф потока данных.

> В ЛР2 в тексте нет отдельного явного блока «Задание: ...» в том же шаблонном формате, как в ЛР1. Для практического выполнения принимается стандартная трактовка: применить критерии покрытия к выбранной функции/модулю и оформить тесты + графы.

## 3) Операционное ТЗ для Codex (как именно выполнять)

### 3.1 Общие правила
1. Не менять смысл исходной методички.
2. Если пользователь не дал объект тестирования — предложить 1 вариант из текущего проекта и зафиксировать его.
3. Для каждого тест-кейса обязательно указывать:
   - входные данные,
   - ожидаемый результат.
4. Все выводы делать проверяемыми (с таблицами).

### 3.2 Что должно быть в финальном отчете
Сформировать один итоговый файл, например `Results/LR1_LR2_Testing.md`, со следующей структурой:

1. Титульный блок (кратко).
2. Цель работы.
3. ЛР1:
   - Ч1: ТЗ на поле, КЭ (таблица), графическое представление, тест-кейсы.
   - Ч2: таблица границ + тесты на границах.
   - Ч3: причины/следствия, функциональная диаграмма, таблица решений, тест-кейсы.
   - Ч4: S&T-диаграмма + маршруты тестирования.
   - Ч5: попарные тесты + анализ (полный перебор vs pairwise).
4. ЛР2:
   - исходный код выбранной функции/фрагмента;
   - тесты на покрытие операторов;
   - тесты на покрытие решений;
   - тесты на покрытие условий;
   - тесты на покрытие решений+условий;
   - тесты на комбинаторное покрытие условий;
   - CFG (и, при возможности, data-flow) + сопоставление тестов путям.
5. Заключение (что покрыто, ограничения, какие риски остаются).

---

## 4) Требования к оформлению таблиц

### Для ЛР1 Ч1 (КЭ)
- Таблица КЭ: `Критерий | Правильные КЭ | Неправильные КЭ`.
- Таблица тестов: `№ | КЭ | Входные данные | Ожидаемый результат`.

### Для ЛР1 Ч2 (границы)
- Таблица границ: `Тип границы | Код | Описание | Значение/диапазон`.
- Таблица тестов: `№ | Граница (код) | Входные данные | Ожидаемый результат`.

### Для ЛР1 Ч3 (решения)
- Таблица причин/следствий: `ID | Формулировка`.
- Таблица решений: столбцы-комбинации причин и строки следствий.
- Таблица тестов: `№ | Входные данные | Ожидаемый результат | Комбинация`.

### Для ЛР2
Для каждого критерия покрытия — отдельная таблица:
`№ | Покрываемые элементы | Входные данные | Ожидаемый результат`.

Для CFG:
- список вершин/ребер,
- список путей,
- таблица соответствия `Тест -> путь -> покрытие`.

---

## 5) Критерии приемки (Definition of Done)
Работа считается выполненной, если:
1. Закрыты **все 5 частей ЛР1** и **все 6 частей ЛР2**.
2. Для каждой части есть артефакты (таблицы/диаграммы/тесты), а не только текстовое описание.
3. Во всех тест-кейсах присутствуют входные данные и ожидаемый результат.
4. По ЛР2 явно показано, какие элементы покрытия достигнуты каждым набором тестов.
5. Нет внутренних противоречий между ТЗ, таблицами и тест-кейсами.
