# Project Self-Learning Roadmap

Этот документ нужен для самообучения: чтобы ты не просто читал код, а постепенно начал **свободно понимать проект**, его архитектуру, термины, технологии и логику работы.

## 1. Как пользоваться этим документом

Работай по следующему принципу:

1. Сначала выучи словарь терминов и сокращений.
2. Потом разберись в общей карте проекта.
3. Затем пройди roadmap по этапам.
4. На каждом этапе обязательно открывай указанные файлы и проговаривай вслух, что ты понял.
5. Не переходи к следующему уровню, пока не можешь своими словами объяснить предыдущий.

## 2. Словарь терминов проекта

### Архитектура и организация кода

- **Репозиторий**: весь проект целиком, включая код, SQL, документацию, тестовые материалы и отчёты.
- **Микросервис**: отдельное приложение с собственной ответственностью. В проекте это `auth-service`, `deal-service`, `payment-service`.
- **Сервис**: приложение, выполняющее определённую бизнес-задачу.
- **Модуль**: отдельная часть кода с собственным `go.mod`.
- **Workspace (`go.work`)**: файл, который объединяет несколько Go-модулей в одно рабочее пространство.
- **Слой**: уровень архитектуры. Например: `transport`, `service`, `repository`.
- **Bounded context**: граница предметной области. В проекте это условно `auth`, `deal`, `payment`.
- **Shared library / common module**: общий код для нескольких сервисов. Здесь это `platform-common`.
- **Runtime architecture**: как система реально работает во время запуска.
- **Legacy / черновая ветка**: старая или неполная реализация, которая уже не является основной.

### Внутренняя структура сервиса

- **`cmd`**: точка входа в сервис, где всё собирается и запускается.
- **`transport`**: слой приёма внешних запросов или событий.
- **`service`**: слой бизнес-логики.
- **`repository`**: слой доступа к базе данных.
- **`integration client`**: код для вызова другого сервиса по HTTP или другому протоколу.
- **Middleware**: промежуточная логика между HTTP-запросом и обработчиком. Например: логирование, recovery, trace-id, auth.
- **Handler**: HTTP-обработчик конкретного маршрута.

### Данные и база

- **PostgreSQL**: основная база данных проекта.
- **Schema / схема БД**: логическая группа таблиц внутри одной PostgreSQL. Здесь есть `auth`, `deal`, `payment`.
- **Migration / миграция**: управляемое изменение структуры БД.
- **Seed data**: тестовые или стартовые данные для локального запуска.
- **Constraint**: ограничение в БД, которое защищает данные.
- **Transaction / транзакция**: группа операций с БД, выполняемая как единое целое.
- **Idempotency / идемпотентность**: повторный одинаковый запрос не должен ломать состояние и не должен создавать лишние дубли.

### Сетевое взаимодействие

- **HTTP API**: интерфейс, через который сервисы и клиенты обмениваются запросами.
- **Endpoint**: конкретный URL и метод, например `POST /api/bookings`.
- **REST**: стиль построения HTTP API.
- **Internal endpoint**: внутренний endpoint для межсервисного взаимодействия, не для внешнего клиента.
- **Swagger / OpenAPI**: описание API в машиночитаемом и человекочитаемом виде.

### Безопасность

- **JWT**: токен, который используется для авторизации пользователя.
- **Claim**: поле внутри JWT. Например: `sub`, `role`, `email`.
- **Role / роль**: тип пользователя. В проекте есть `client` и `admin`.
- **Authorization**: проверка, что пользователь имеет право выполнять действие.
- **Authentication**: проверка, кто именно пользователь.
- **Shared secret**: общий секрет для подписи и проверки JWT.
- **Provisioning**: создание и подготовка пользователей, ролей, доступов и окружения.

### Асинхронная архитектура

- **Event / событие**: сообщение о том, что что-то произошло. Например `payment.completed`.
- **Message broker**: посредник для обмена событиями между сервисами. Здесь это `NATS`.
- **Publish**: отправить событие.
- **Subscribe**: подписаться на событие.
- **Consumer**: обработчик полученного события.
- **Async flow**: сценарий, который продолжается не через прямой HTTP-ответ, а через события.
- **Durable subscription**: подписка, которая не теряет сообщения при перезапуске.
- **Retry**: повторная попытка обработки.
- **Outbox / Inbox**: паттерны повышения надёжности доставки событий.

### Наблюдаемость и эксплуатация

- **Health endpoint**: endpoint для проверки состояния сервиса.
- **Readiness**: сервис готов принимать трафик.
- **Liveness**: сервис жив и не завис.
- **Metrics**: числовые показатели работы системы.
- **Prometheus**: система сбора метрик.
- **Structured logging**: логирование в структурированном формате.
- **Trace ID**: идентификатор запроса, чтобы отслеживать его путь между сервисами.
- **Recovery middleware**: middleware, который ловит panic и не даёт сервису упасть на одном запросе.

### Предметная область проекта

- **Sanatorium / санаторий**: объект размещения и лечения в системе.
- **Catalog**: список санаториев с фильтрацией и сортировкой.
- **Booking / бронирование**: запись о брони санатория клиентом.
- **Contract / договор**: договор на обслуживание/заселение.
- **Invoice / инвойс / счёт**: счёт на оплату, связанный с договором.
- **Payment / платёж**: факт оплаты счёта.
- **Capacity / вместимость**: сколько гостей можно разместить в санатории или слоте.

## 3. Список сокращений и обозначений

## 4. Какие знания нужны, чтобы понимать проект свободно

Чтобы уверенно разбираться в проекте, тебе нужно освоить 5 блоков:

1. **Общая архитектура проекта**
2. **Go-структура сервисов**
3. **HTTP API и JWT**
4. **PostgreSQL и схема данных**
5. **Асинхронные события через NATS**

Если один блок выпадает, проект будет восприниматься фрагментарно.

## 5. Roadmap самообучения

### Этап 1. Освоить карту проекта

Цель: понять, из чего состоит репозиторий и что является основной системой.

Что изучать:

- `docs/project-understanding-guide.md`
- `docs/project-readiness-audit.md`
- `src/go-microservices/docker-compose.yml`
- структуру каталогов в `src`

Что нужно уметь объяснить после этапа:

- какая часть проекта основная;
- какие сервисы существуют;
- какие папки являются кодом, а какие документацией и артефактами;
- почему `.NET`-ветка сейчас не является главным объектом изучения.

Мини-практика:

- нарисуй от руки дерево репозитория;
- подпиши, какие каталоги являются runtime-кодом, а какие нет;
- попробуй своими словами объяснить проект за 2 минуты.

### Этап 2. Освоить технологический стек

Цель: понять, какими технологиями собран проект и зачем каждая нужна.

Что изучать:

- Go workspace: `src/go-microservices/go.work`
- `docker-compose.yml`
- `sql/postgres`
- `platform-common`

Что нужно уметь объяснить:

- зачем нужен Docker Compose;
- зачем используется PostgreSQL;
- зачем нужен NATS;
- зачем нужен `platform-common`;
- что делает `go.work`.

Мини-практика:

- составь таблицу “технология -> зачем она в проекте”;
- для каждой технологии ответь на вопрос: “что перестанет работать, если её убрать?”.

### Этап 3. Освоить устройство одного Go-сервиса

Цель: понять шаблон, по которому устроены сервисы.

Что изучать:

- `auth-service/cmd/auth-service/main.go`
- `auth-service/internal/transport/http/handlers.go`
- `auth-service/internal/service/auth.go`
- `auth-service/internal/repository/postgres.go`

Что нужно уметь объяснить:

- как запрос проходит от HTTP до БД;
- где находится бизнес-логика;
- где создаётся JWT;
- где сервис конфигурируется и запускается.

Мини-практика:

- проследи руками путь запроса регистрации;
- проследи руками путь запроса логина;
- выпиши цепочку `handler -> service -> repository -> database`.

### Этап 4. Освоить `deal-service`

Цель: понять самый важный и самый “толстый” сервис проекта.

Что изучать:

- `deal-service/internal/transport/http/handlers.go`
- `deal-service/internal/service/booking.go`
- `deal-service/internal/service/deal.go`
- `deal-service/internal/repository/booking_catalog_postgres.go`
- `deal-service/internal/repository/postgres.go`

Что нужно уметь объяснить:

- как работает каталог;
- как создаётся бронирование;
- как создаётся договор;
- как `deal-service` общается с `auth-service` и `payment-service`.

Мини-практика:

- найди в коде все публичные и защищённые endpoint'ы;
- выпиши, какие use-case живут в `deal-service`;
- объясни разницу между booking flow и contract flow.

### Этап 5. Освоить `payment-service`

Цель: понять логику счета, оплаты и событий.

Что изучать:

- `payment-service/internal/transport/http/handlers.go`
- `payment-service/internal/service/payment.go`
- `payment-service/internal/repository/postgres.go`

Что нужно уметь объяснить:

- как создаётся invoice;
- как проходит платёж;
- зачем нужен `Idempotency-Key`;
- как публикуется `payment.completed`.

Мини-практика:

- опиши разницу между `invoice` и `payment`;
- объясни, почему событие публикуется после изменения БД;
- объясни, почему идемпотентность важна для платежей.

### Этап 6. Освоить базу данных

Цель: понять, какие сущности существуют и как они связаны.

Что изучать:

- `sql/postgres/00_init_schemas.sql`
- `sql/postgres/01_booking_catalog.sql`

Что нужно уметь объяснить:

- какие схемы есть в БД;
- какие таблицы к какой предметной области относятся;
- как связаны `bookings`, `contracts`, `invoices`, `payments`;
- какие ограничения защищают данные.

Мини-практика:

- перепиши ER-модель своими словами;
- объясни, почему `payment` и `invoice` не одно и то же;
- найди `CHECK`, `UNIQUE`, `FOREIGN KEY` и объясни их роль.

### Этап 7. Освоить события и интеграции

Цель: понять, где проект синхронный, а где асинхронный.

Что изучать:

- `platform-common/pkg/events/nats.go`
- `deal-service/internal/transport/events/payment_completed_consumer.go`
- integration clients в `deal-service`

Что нужно уметь объяснить:

- что идёт по HTTP, а что через NATS;
- как событие `payment.completed` влияет на `deal-service`;
- какие риски есть у async-подхода в текущем проекте.

Мини-практика:

- нарисуй схему “кто публикует и кто слушает”;
- сравни синхронный вызов `CreateInvoice` и асинхронный `payment.completed`;
- объясни, в чём проблема отсутствия durable delivery.

### Этап 8. Освоить эксплуатационную сторону

Цель: понять, как проект живёт в среде запуска.

Что изучать:

- health endpoints
- metrics
- trace-id
- logging middleware
- Docker Compose

Что нужно уметь объяснить:

- как понять, что сервис жив;
- как понять, что он готов;
- как искать ошибку по логам;
- зачем нужен trace-id;
- как сервисы запускаются вместе.

Мини-практика:

- выпиши список operational endpoints;
- объясни разницу между `live` и `ready`;
- объясни, какие зависимости нужны каждому сервису.

## 8. Практический способ учиться быстрее

Используй цикл из 4 шагов:

1. **Читаю**: открываю файл и пытаюсь понять общий смысл.
2. **Пересказываю**: объясняю своими словами, что делает этот файл.
3. **Связываю**: отвечаю, как он связан с другими файлами и сервисами.
4. **Проверяю**: смотрю, не ошибся ли я, сверяясь с документацией и схемами.

Если просто читать код без пересказа, понимание будет поверхностным.

## 10. Твоя учебная цель

Твоя цель не в том, чтобы “запомнить файлы”.

Твоя цель:

- видеть архитектуру проекта целиком;
- понимать, зачем нужен каждый сервис;
- уметь проследить любой ключевой сценарий от запроса до БД и обратно;
- уметь объяснить сильные и слабые стороны проекта;
- быть готовым самостоятельно дорабатывать систему.

Когда ты сможешь объяснить всё это без шпаргалки, проект станет для тебя действительно понятным.
