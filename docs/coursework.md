# Курсовая работа

**Тема:** Конструирование программных средств управления и учета сделок информационной системы малого гостиничного бизнеса.

## 1. Анализ требований

### 1.1 Контекст и заинтересованные стороны
Система предназначена для малого гостиничного бизнеса и поддерживает процессы бронирования, учета сделок (договора, счета, оплаты), управления номерным фондом и формирования отчетности. Основные пользователи:
- администратор ресепшена;
- менеджер по продажам/бронированию;
- бухгалтер;
- руководитель (для отчетов и аналитики);
- гость (через внешний интерфейс бронирования, при необходимости).

### 1.2 Функциональные требования
1. Регистрация и аутентификация пользователей с разграничением ролей (администратор, бухгалтер, менеджер).
2. Управление справочниками (номера, категории, тарифы, дополнительные услуги).
3. Создание, изменение и отмена бронирований.
4. Ведение сделки (договор/заказ): состав услуги, стоимость, сроки проживания.
5. Управление заселением и выселением гостей.
6. Формирование счетов и актов по сделкам.
7. Учет оплат (наличные/безналичные), возвраты.
8. Контроль задолженности по сделкам.
9. Генерация отчетов (выручка, загрузка, задолженность, динамика продаж).
10. Поиск и фильтрация по клиентам, сделкам и периодам.
11. Ведение журнала действий пользователей (аудит).
12. Экспорт данных (например, CSV/Excel) для бухгалтерии.
13. Интеграция с внешней отчетностью/хранилищем данных (через экспорт или события).

### 1.3 Нефункциональные требования
- **Производительность:** основные операции (поиск, создание брони, формирование счета) выполняются за 1–2 сек при стандартной нагрузке.
- **Надежность:** отказ одного сервиса не должен приводить к отказу всей системы; устойчивость за счет изоляции сервисов.
- **Масштабируемость:** возможность независимого горизонтального масштабирования сервисов.
- **Безопасность:** RBAC, хранение паролей по стандарту (хеширование), TLS для внешних интерфейсов, аудит действий.
- **Наблюдаемость:** мониторинг состояния компонентов, метрики, централизованное логирование.
- **Сопровождаемость:** модульная структура, четкие контракты между сервисами.
- **Тестируемость:** наличие модульных и интеграционных тестов, автоматическая проверка критичных бизнес-процессов.
- **Портативность:** работа под управлением Windows 10+; поддержка запуска из Visual Studio 2019/2022 или VSCode.

## 2. Выбор архитектурного стиля

### 2.1 Рассмотренные варианты
- **Монолит:** проще в разработке, но хуже масштабируется и не соответствует требованию о распределенной архитектуре.
- **Классическая SOA:** избыточна для малого бизнеса, повышает сложность интеграций.
- **Микросервисный стиль:** подходит для распределенного решения, поддерживает независимое развертывание модулей и асинхронные сценарии.

### 2.2 Выбранный стиль
**Микросервисная архитектура с событийным взаимодействием (event-driven) между внутренними компонентами.**

**Обоснование выбора:**
- удовлетворяет требованию наличия 3 микросервисов;
- повышает изолированность компонентов и масштабируемость;
- поддерживает синхронные (REST/gRPC) и асинхронные (сообщения) взаимодействия;
- упрощает мониторинг состояния компонентов за счет изолированных health-check и метрик.

## 3. Проектирование программного обеспечения

### 3.1 Состав микросервисов
1. **Auth Service** — управление пользователями, ролями и токенами доступа.
2. **Deal Service** — управление бронированиями и сделками (заказы, условия, статусы, заселение/выселение).
3. **Payment Service** — выставление счетов, учет оплат и задолженности.

### 3.2 Модульность
Каждый микросервис разбивается на отдельные модули (проекты) внутри решения:
- `*.Api` — HTTP/gRPC API и контроллеры;
- `*.Application` — бизнес-логика и сценарии;
- `*.Domain` — доменные сущности и правила;
- `*.Infrastructure` — доступ к БД, интеграции, брокер сообщений;
- `*.Contracts` — DTO и контракты для взаимодействия.

### 3.3 Хранение данных
- **Изолированная БД** для проекта (отдельно от учебной БД/ХД).
- Подход **database-per-service**: отдельные базы/схемы для каждого сервиса (`AuthDB`, `DealDB`, `PaymentDB`).
- Внешнее хранилище данных (DWH) подключается через экспорт или асинхронные события (ETL/CDC).

### 3.4 Взаимодействие компонентов
- **Синхронно:** REST/gRPC запросы через API Gateway для операций пользователя (создание брони, оплата, получение отчета).
- **Асинхронно:** обмен событиями через брокер сообщений (например, `DealCreated`, `PaymentCompleted`, `GuestCheckedIn`).

### 3.5 Мониторинг и наблюдаемость
- Health-check endpoints для каждого сервиса.
- Метрики (Prometheus) и визуализация (Grafana).
- Централизованные логи (например, Seq/ELK).
- Трассировка запросов (OpenTelemetry).

## 4. Перечень функций системы

1. Аутентификация и управление пользователями.
2. Управление ролями и правами доступа.
3. Ведение справочников номеров и тарифов.
4. Создание и изменение бронирований.
5. Отмена и перенос бронирований.
6. Управление статусами заезда/выезда.
7. Формирование сделок (договоров/заказов) и условий проживания.
8. Формирование счетов по сделкам.
9. Учет оплат и возвратов.
10. Контроль задолженности по сделкам.
11. Генерация финансовых и операционных отчетов.
12. Экспорт данных в табличные форматы.
13. Журналирование действий пользователей.
14. Отправка событий в систему аналитики/хранилище данных.

## 5. Тестирование и отладка

- Модульные тесты для доменной логики (расчеты стоимости, статусы сделки, правила отмены).
- Интеграционные тесты для API (создание брони, оплата, формирование отчета).
- Нагрузочное тестирование ключевых операций (поиск, отчетность).
- Проверка работоспособности на Windows 10+ в среде Visual Studio 2019/2022 или VSCode.

## 6. Заключение

Предложенная архитектура удовлетворяет требованиям курсовой работы: система многомодульна, поддерживает синхронные и асинхронные взаимодействия, обеспечивает мониторинг состояния компонентов и пригодна для эксплуатации в среде Windows 10+.
