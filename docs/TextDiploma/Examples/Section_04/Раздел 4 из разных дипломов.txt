4 Контроль качества программного обеспечения
4.1 Выбор режима тестирования и метрик кода
Тестирование – важная часть процесса разработки программного обеспечения, оно проводится на разных этапах проекта. Тестирование программного обеспечения – это метод оценки функциональности программного обеспечения. В ходе этого процесса проверяется, соответствует ли фактическое программное обеспечение ожидаемым требованиям, и обеспечивается отсутствие ошибок в программном обеспечении. Целью тестирования является выявление ошибок, сбоев или недостающих требований по сравнению с фактическими требованиями. В основном оно направлено на измерение спецификации, функциональности и производительности программного обеспечения или приложения [16].
Наиболее распространены такие виды тестирования, как функциональное, модульное, интеграционное и нагрузочное.
Функциональное тестирование проверяет систему на соответствие функциональным требованиям и спецификациям, гарантирует их должное удовлетворение, фокусируется на моделировании реального использования системы. Модульное тестирование направлено на проверку отдельных небольших частей приложения, которые можно исследовать изолированно от других подобных частей [17].
Интеграционное тестирование – это метод проверки взаимодействия различных модулей или компонентов программного приложения друг с другом. Нагрузочное тестирование – это вид тестирования производительности, который имитирует реальную нагрузку на систему или приложение, чтобы проверить, как они работают в условиях стресса [18].
Для полного охвата взаимодействия пользователя с приложением необходимо написать тест-кейсы с детальным описанием действий для проверки конкретных функций, основанные на пользовательских сценариях.
Главным основанием для выбора функционального тестирования в качестве метода ручного тестирования послужило большое внимание требований к программному продукту, обращенных на функциональные аспекты. В качестве автоматизированного тестирования было выбрано модульное тестирование, юнит тесты. Данный метод позволяет эффективно и в короткий срок провести проверку отдельных компонентов системы.
Возросшая сложность современных программных приложений также усложняет задачу обеспечения надёжности и удобства сопровождения кода. Метрики кода – это набор программных показателей, которые дают лучшее представление о коде, позволяют понять, какие типы или методы следует переработать или протестировать более тщательно [19].
Для оценки качества кода разработанного приложения-агрегатора для анализа условий и подбора ипотечных предложений были выбраны такие метрики, как цикломатическая сложность, метрика Холстеда, а также метрика количества строк кода (SLOC).
Цикломатическая сложность – мера структурной сложности кода. Она рассчитывается путем подсчета количества различных путей кода в потоке программы. Программные метрики Холстеда – это набор показателей, основанных на количестве различных операторов и операндов в программе, используется для оценки усилий, необходимых для разработки и поддержки программы. Метрика SLOC используется для количественной оценки размера и сложности программного обеспечения. Она измеряет количество строк в исходном коде программы, включая комментарии и пустые строки [20, 21].

4.3.1 Цикломатическая сложность
Цикломатическая сложность является одной из метрик программного обеспечения, которая позволяет количественно оценить логическую сложность программы. Для вычисления цикломатической сложности на примере функции расчета ипотечных условий был использован граф потока управления программы, представленный на рисунке 31.
 
Рисунок 31 – Управляющий граф прецедента «Расчет ипотечных параметров»
Для вычисления цикломатической сложности Z(G) применяется следующая формула:
Z(G)=e-v+2p,
где     e – число дуг ориентированного графа G; 
v – число вершин; 
p – число компонентов связности графа.
Функциональный граф, построенный для алгоритма расчета ипотечных параметров, состоит из 8 дуг и 8 вершин. Для простой программы, или подпрограммы, или метода число компонентов связанности графа всегда равно 1. На основании этих данных была определена циклическая сложность метода. Ее значения получилось равным 2.
Данный результат позволяет говорить, что метод обладает средней структурной сложностью.
4.3.2 Метрика Холстеда
Для расчета метрик Холстеда рассматривается функция Calculate компонента HomeController. Функция реализует расчет основных ипотечных параметров, а также подбор банковских предложений, соответствующих введенным данным пользователя. Код данной функции представлен в листинге 1.
Листинг 1 – Исходный код функции Calculate
[HttpPost]
public async Task<IActionResult> Calculate(HomeLoan model)
{
    decimal loanAmount = model.HomePrice – model.Payment;
    decimal monthlyInterestRate = model.Rate / 1200;
    int term = model.Term;
    model.MonthlyPayment = loanAmount * (monthlyInterestRate * (decimal)Math.Pow((double)(1 + monthlyInterestRate), term)) /
                                           
((decimal)Math.Pow((double)(1 + monthlyInterestRate), term) – 1);
    Model.TotalPayment = model.MonthlyPayment * term;
    Model.Overpayment = model.TotalPayment – loanAmount;
    model.LoanAmount = loanAmount;
  
    model.BankOffers = await _context.Offer
        .Where(o => o.InterestRate <= model.Rate && 
o.LoanAmountMax >= loanAmount)
        .ToListAsync();
    Return View(model);
}
Для вычисления данной метрики требуется посчитать число вхождения уникальных операторов (N1) и операндов (N2), а также общее количество операторов (n1) и операндов (n2).

4.1	Обоснование выбора модели тестирования и метрик кода
В процессе разработки серверной части логистического сервиса особое внимание уделяется обеспечению качества программного кода. Комплексный подход к тестированию и анализу качеству кода позволяет гарантировать надежность, безопасность и эффективность функционирования частей микросервисной архитектуры. 
Основные виды тестирования серверной части включают:
	модульное тестирование – проверка корректности работы отдельных компонентов и модулей системы в изоляции [15];
	интеграционного тестирование – верификация взаимодействия между микросервисами и компонентами системы [16];
	тестирование производительности – оценка скорости обработки запросов, нагрузки на систему и масштабируемости;
	функциональное тестирование – проверка соответствия реализованных функций заявленным требованиям [17];
Метрики качества кода играют важную роль в оценке надежности и эффективности серверной части логистического сервиса. В рамках данной выпускной квалификационной работы были выбраны следующие метрики: SLOC и метрика Холстеда. 
SLOC (Source Lines of Code) – метрика, измеряющая количество строк исходного кода приложения. Она позволяет:
	оценить объем разработки;
	прогнозировать трудозатраты на поддержку;
	выявлять избыточную сложность кода. 
Метрика Холстеда – комплексная метрика, основанная на анализе операторов и операндов в коде. Она обеспечивает:
	оценку сложности программы;
	прогнозирование трудоемкости разработки;
	анализ потенциальной надежности системы [18]. 
Выбор данных метрик обусловлен их способностью предоставить объективную оценку качества кода микросервисов, выявить потенциальные проблемы в архитектуре и обеспечить контроль над сложностью системы при ее развитии. Регулярный мониторинг выбранных метрик позволяет поддерживать высокий уровень качества серверной части логистического сервиса. 
Комплексный подход к тестированию и анализу качества кода обеспечивает соответствие разрабатываемого программного обеспечения предъявляемым требованиям и гарантирует его надежную работу в условиях реальной эксплуатации.

Длинна программы рассчитывается по формуле:
N = N1 + N2,
где N1 – общее число операторов в программе;
N – общее число операндов в программе.
Таким образом, на основе данных из таблицы 11, длина программы:
N = 111 + 93 = 203. 
Словарь программы рассчитывается по формуле:
n = n1 + n2,
где n1 – число уникальных операторов программы;
n2 – число уникальных операндов программы. 
Словарь программы на основе данных их таблицы 12:
n = 20 + 17 = 37.
Расчет вычислительная сложность программы находится по формуле:
HV = N * log2n,
где N – длина программы;
n – словарь программы;
Таким образом, на основе ранее рассчитанных показателей N и n, вычислительная сложность программы составит:
HV = 203 * log237 = 1057,427. 
 Кроме того, был также рассчитан комплексный показатель качества кода – Maintainability Index. Этот показатель помогает оценить, насколько легко поддерживать и модифицировать код. Высокий индекс означает более легко поддерживаемый код. Расчет показателя происходит по формуле:
MI = 171 - 5.2 * ln(HV) - 0.23 * CC – 16.2 * ln(LOC),
где HV – вычислительная сложность кода;
СС – цикломатическая сложность кода;
LOC – количество строк кода. 
Показатель СС, для данного кода равен 4, количество строк кода насчитывает 105 строк, а вычислительная сложность была рассчитана выше и составляет 1057,427. 
Таким образом комплексный показатель:
MI = 171 - 5.2 * ln(1057,427) - 0.23 * 4 – 16.2 * ln(105) = 58,477. 
Результаты анализа находятся в пределах нормы, что соответствует среднему уровню сопровождаемости и сложности программного кода, а также соответствуют установленным критериям оценки качества кода. Это свидетельствует о невысокой сложности и прозрачности кодовой базы для разработчиков и простоты ее модификации.
