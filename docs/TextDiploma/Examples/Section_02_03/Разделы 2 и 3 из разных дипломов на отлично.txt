Проектирование программных средств для осознанного планирования жизни
2.1 Архитектура приложения
Архитектура программных средств для осознанного планирования жизни представляет собой микросервисную архитектуру в сочетании с шаблоном проектирования MVC. Контроллер и модель реализованы внутри каждого микросервиса, они отвечают за хранение и обработку данных, а также обрабатывают действия пользователя, соединяют модели и представления. Клиентское приложение обеспечивает отображение данных и служит основным интерфейсом для получения команд от пользователей, которые впоследствии направляются на сервер посредством REST API [6]. 
На рисунке 10 изображена диаграмма развертывания сервиса, которая отображает архитектуру программных средств для осознанного планирования жизни.
 
Рисунок 10 – Архитектура программных средств для осознанного планирования жизни
Серверная часть приложения представлена в виде трех независимых микросервисов, каждый из которых имеет свой функционал, отвечающий за определенную часть целостного приложения, каждый сервис взаимодействует с отдельной базой данных. Микросервисы могут обращаться друг к другу через API для совместной обработки данных. Таким образом, серверная часть выступает промежуточным слоем между запрашиваемыми ресурсами и базами данных [7].
После получение данных микросервисом от БД, он их анализирует, создает ответ в виде JSON и отправляет его назад клиенту для визуализации. Этот процесс подчеркивает адаптивность и возможность масштабирования системы, обеспечивая автономную и эффективную работу каждого из элементов.

2.2.2 Логические модели баз данных
В разрабатываемых программных средствах для осознанного планирования жизни была выделена разработка трех баз данных: «Авторизация», «Планирование», «Отчеты».
Разделение было произведено по следующим причинам:
	разделяя данные по функциональным областям, удобнее управлять их хранением и обработкой;
	использование нескольких баз данных позволяет сбалансировать нагрузку;
	разделение данных по нескольким БД позволяет эффективнее организовывать резервное копирование и восстановление данных [9].

2.3 Проектирование пользовательского интерфейса
Интерфейс web-приложения должен отвечать всем функциональным требованиям, а также должен быть удобным в использовании.
Проектируемый интерфейс включает в себя страницы входа, регистрации и главной страницы с навигационным меню по приложению. На главной странице должны располагаться вкладки с различным содержимым, таким как список имеющихся в системе задач, настройки системы, календари с запланированным и фактическим затраченным временем, а также графики с анализом деятельности человека. 
Кроме вкладок и страниц в приложении должны быть реализованы модальные окна добавления таких объектов, как «Задача», «Сфера жизни», «Направление», «Проект», «Цель», «Запланированное время», модальные окна детального просмотра объектов системы, созданных пользователем.
Каждая составляющая интерфейса будет реализована с использованием компонентов библиотеки ReactJS, контролирующих свое состояние и функциональность. Применение архитектуры, основанной на компонентах, упрощает расширение приложения и использование кода повторно [10].

3 Реализация программных средств для осознанного планирования жизни
3.1 Реализация базы данных
Физическая модель базы данных – это самое детальное описание ее реализации, включая типы данных, индексы, физическое размещение файлов.
На основе концептуальной модели базы данных и логических были построены физические модели базы данных для баз, использующихся в разрабатываемом программном средстве [11].
Физическая модели баз данных «Планирование», «Авторизация и Отчеты» были построены в среде Visual Paradigm 17.0 в нотации IDEF1X, они представлены на рисунках В.7, В.8 и В.9 в приложении В.
База данных, соответствующая разработанной физической модели, была развернута с помощью механизма миграций.
Механизм миграций необходим для версионирования и синхронизации структуры БД. 
3.2 Реализация серверной части приложения
Серверная часть программных средств для осознанного планирования жизни разрабатывалась в среде программирования JetBrains Rider, на языке программирования C#. Код серверной части приложения приведен в приложении Г.
Серверная часть состоит из трех микросервисов. Каждый микросервис содержит модели, отражающие структуру баз данных, а также сериализаторы для преобразования данных моделей в удобный формат для обмена между сервером и клиентом. Кроме того, реализованы специализированные классы для миграции данных, что способствует адаптации и поддержке структуры данных в соответствии с изменяющимися требованиями. Эти классы важны для автоматизации процессов обновления схем данных, что помогает минимизировать возможные риски потери информации при внедрении нововведений и облегчает развертывание обновлений в операционной среде.
Разработанные сервисы предоставляют REST API для взаимодействия. Реализованные методы описаны в таблице 5.

2.1 Архитектура приложения
Разработка архитектуры приложения представляет собой процесс проектирования структуры, компонентов и взаимодействия всех частей, обрабатывающих данные, выполняющих бизнес-логику и обеспечивающих взаимодействие с клиентом и другими системами. При разработке архитектуры важно учитывать масштабируемость, технологии, безопасность и производительность приложения, чтобы избежать основных трудностей, возникающих в процессе разработки [4].
Монолитная архитектура – это методология проектирования программного обеспечения, которая объединяет все компоненты приложения в единое неразделимое целое. При такой архитектуре пользовательский интерфейс, бизнес-логика и уровни доступа к данным создаются, используются и поддерживаются как единое целое. Ключевыми компонентами монолитной архитектуры являются:
	пользовательский интерфейс (UI);
	логика приложения;
	уровень доступа к данным;
	база данных [5].
Компонентами приложения с шаблоном проектирования MVC являются модель, представление и контроллер. Контроллер – это компонент, который обрабатывает входящий запрос. Компоненты контроллера выполняют несколько действий для обработки запроса. Компонент контроллера создаёт модель, необходимую для представления. Модель – это компонент, который в основном содержит классы, используемые для хранения данных предметной области или, можно сказать, бизнес-данных. В шаблоне проектирования MVC компонент модели также содержит необходимую логику для получения данных из базы данных.
Как только контроллер создает модель, он выбирает представление для отображения данных домена или модели. При выборе представления контроллер также передает данные модели. В шаблоне проектирования MVC единственной обязанностью представления является отображение данных модели. Таким образом, в MVC представление – это компонент, отвечающий за создание необходимого HTML-кода для отображения данных модели. Как только представление создает HTML-код, этот HTML-код отправляется клиенту по сети, который изначально отправил запрос [6].
Согласно выделенным требованиям к программных средствам, была разработана диаграмма развертывания, отражающая архитектуру приложения-агрегатора для анализа условий и подбора ипотечных предложений. Разработанная диаграмма представлена на рисунке 11.

2.2.2 Логическая модель базы данных
Логическая модель данных – это расширение концептуальной модели данных. Она включает в себя все сущности, атрибуты, ключи и взаимосвязи, которые представляют бизнес-информацию и определяют бизнес-правила [8]. Логическая модель является следующим этапом при проектировании базы данных, она сложнее концептуальной модели, так как возникает необходимость определить типы столбцов каждой сущности.
Концептуальная модель данных строится на основе нотации сущность-связь, включающей описание сущностей, атрибутов и связей между сущностями. В ходе логического проектирования реляционной базы данных (РБД) элементы концептуальной модели «сущность-связь» отображаются в элементы реляционной модели (отношения, атрибуты отношений, связи между отношениями).
Наиболее часто на этапе логического проектирования РБД применяется предметно-ориентированный подход, при котором схема базы данных строится для конкретной предметной области с учетом правил логического проектирования. При этом сущностям концептуальной модели «сущность-связь» в реляционной модели ставится в соответствие отношение, каждому свойству сущности – атрибут отношения, связи между сущностями – связь между отношениями.

2.4 Проектирование пользовательского интерфейса
Для реализации продукта, способного заинтересовать большее количество пользователей, необходимо спроектировать информативный и удобный дизайн пользовательского интерфейса. Дизайн пользовательского интерфейса – это часть процесса разработки программного обеспечения, которая создаётся после глубокого изучения пользователей и особенностей предметной области [10].
В соответствии с результатами анализа предметной области и выявленными требованиями к разрабатываемому продукту был спроектирован пользовательский интерфейс. 
Интерфейс веб-приложения должен быть понятным и удобным в использовании для пользователей с разным уровнем вовлеченности в предметную область, что позволит увеличить количество потенциальных пользователей.
Проектируемый интерфейс включает в себя несколько страниц:
	главная страница калькулятора;
	страница справочной информации;
	страница регистрации в приложении;
	страница входа в приложение;
	страница избранных предложений пользователя;
	страница управления аккаунтом.
Страницы избранного и управления аккаунтом доступны только для авторизованных пользователей.
Каждая страница сопровождается навигационным меню по приложению в верхней части. Данной расположение является типичным для данной предметной области, что позволяет ускорит адаптацию новых пользователей. На главной странице должен располагаться ипотечный калькулятор, после взаимодействия с которым пользователь сможет увидеть результаты расчета, график платежей, а также подобранные по введенным условиям ипотечные предложения. 
Страница справочной информации должна предоставлять информацию о возможностях использования калькулятора, а также о сложных терминах предметной области, использующихся при расчете. Избранное пользователя должно содержать сохраненные предложения, заметки к предложениям, а также навигацию для перехода на страницы редактирования заметок, удаления предложения и страницы с подробной информацией о предложении.
Интерфейс будет реализован с использованием представлений, которые являются неотъемлемой частью шаблона проектирования MVC. Представления будут созданы с помощью лаконичного и эффективного языка разметки Razor. В Razor шаблон представления сочетает в себе HTML-страницу с несколькими фрагментами кода. Код во фрагментах, написанный на языке C#, вычисляется при рендеринге представления, и полученная разметка интегрируется в шаблон HTML. Механизм Razor считывает шаблоны представлений из физического местоположения на диске. Путь определяется с помощью поставщика виртуальных путей ASP.NET [11, 12].

База данных, соответствующая разработанной физической модели, была развернута с помощью механизма миграций. Миграции – это функция в Entity Framework, которая позволяет управлять изменениями в схеме базы данных с течением времени. Загрузка данных позволяет предварительно заполнить базу данных набором исходных данных [14]. Для работы с базой данных использовалась среда Microsoft SQL Server Management Studio.
3.2 Реализация основного приложения
Реализация основного приложения проводилась в среде разработки Microsoft Visual Studio 2022, на языке программирования C#, с использованием среды ASP.NET Core и шаблона MVC.
ASP.NET Core является кроссплатформенной средой с открытым исходным кодом для создания современных приложений, оптимизированных для работы в облаке. Приложения ASP.NET Core отличаются простотой и модульностью, а также реализуют встроенную поддержку внедрения зависимостей, что позволяет расширить возможности тестирования и сопровождения. В сочетании с моделью MVC, которая поддерживает создание современных веб-API в дополнение к приложениям на основе представлений, ASP.NET Core представляет собой эффективную платформу для разработки веб-приложений [15].
Общая архитектура приложения использует множество различных представлений MVC и конечных точек URL-адресов. Такой подход позволяет минимизировать количество функций, обрабатываемых на стороне клиента, тем самым разделяя ответственность между сервером и клиентом. При этом отдельные операции на конкретной странице оптимизированы с использованием функций AJAX (асинхронный JavaScript и XML). Для наглядного отображения взаимосвязи между компонентами системы была разработана диаграмма компонентов, представленная на рисунке 16.

3.3 Реализация интерфейса приложения
Общий принцип интерфейса приложения состоит в переключении страниц на главном окне. Это позволяет вести независимую разработку каждой страницы отдельно, не нагружая при этом пространство логики для основного окна. Пользовательский интерфейс должен обеспечивать привычное и удобное представление информации, простое и эффективное выполнение основных функций пользователей. Разработка представлений проводилась с применением разметки Razor, в соответствии с выделенными ранее требованиями к пользовательскому интерфейсу.
В результате были реализованы страницы веб-приложения-агрегатора для анализа условий и подбора ипотечного кредитования:
	страница ипотечного калькулятора;
	страница справочной информации;
	страница регистрации;
	страница авторизации;
	страница «Избранного»;
