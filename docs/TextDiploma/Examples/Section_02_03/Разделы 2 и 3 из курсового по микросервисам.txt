2	Проектирование приложения
2.1	Высокоуровневое проектирование
Для удовлетворения требований к разрабатываемой системе была выбрана микросервисная архитектура. Микросервисная архитектура (MSA, microservices architecture) - это архитектура программного обеспечения, которая состоит из комплекса отдельных модулей (микросервисов), обеспечивающих выполнение независимых функций. Такая распределённая система отличается гибкостью в разработке и широкими возможностями масштабирования [1]. Для отображения выделяемых в системе сервисов составлена диаграмма компонентов (Рисунок 2).

 
Рисунок 2 – Диаграмма компонентов панели клиента
Согласно диаграмме, в системе выделяются следующие сервисы и инфраструктурные компоненты:
	клиентскую часть системы представляет web-приложение, через которое клиент взаимодействует с каталогом и собственными бронированиями, а сотрудник служебного контура - с договорами;
	микросервис аутентификации (auth-service) - управляет регистрацией клиентов, проверкой учётных данных и выдачей JWT-токенов;
	микросервис бронирований и каталога (deal-service) - обеспечивает работу с каталогом санаториев, клиентскими бронированиями и служебными договорами, а также осуществляет межсервисную интеграцию с сервисом оплаты;
	микросервис оплаты (payment-service) - управляет счетами и платежами, обеспечивает идемпотентную обработку платежей и публикует события о завершённых оплатах;
	общий модуль платформы (platform-common) - содержит инфраструктурные утилиты: структурированное логирование, обёртку над NATS, middleware для метрик и трассировки;
	СУБД PostgreSQL - хранилище данных, разделённое на схемы auth, deal и payment по принадлежности к бизнес-доменам;
	брокер сообщений NATS - асинхронная шина доменных событий; используется для потоков booking.confirmed, booking.cancelled и payment.completed.
Выбор указанной архитектуры позволяет в наибольшей степени удовлетворить все требования, предъявленные к системе при анализе: независимое развитие и развёртывание сервисов, устойчивое разделение ответственности по бизнес-доменам и возможность дальнейшего расширения без полной переработки системы.
2.2	Проектирование серверной части
Серверная часть разбита на три микросервиса. При реализации каждого сервиса соблюдался единый подход на основе многослойной архитектуры. Выделены следующие слои:
Transport Layer (слой представления) - HTTP-обработчики, middleware аутентификации и трассировки, структуры запросов и ответов REST API;
Service Layer (слой бизнес-логики) - доменная валидация входных данных, оркестрация операций, работа с событиями;
Repository Layer (слой данных) - SQL-запросы к PostgreSQL, маппинг строк на доменные структуры, транзакции;
Domain Layer (доменный слой) - структуры, отражающие бизнес-сущности (пользователь, санаторий, бронирование, договор, счёт, платёж);
Integration Layer (слой интеграций) - HTTP-клиенты к другим сервисам и продюсеры/подписчики NATS.

2.2.1 Проектирование микросервиса аутентификации (auth-service)
Микросервис аутентификации отвечает за регистрацию клиентов и выдачу JWT-токенов. Структура микросервиса представлена на диаграмме классов (Рисунок 3).
 
Рисунок 3 – Диаграмма классов микросервиса аутентификации (auth-service)
Согласно данной схеме можно выделить следующие составные элементы:
	AuthHandler - основной обработчик HTTP-запросов регистрации и входа, а также точек доступа мониторинга;
	RegisterRequest - структура запроса на регистрацию, содержит email, пароль, полное имя и роль;

	UserRepository - слой данных, инкапсулирует работу с таблицей auth.users, поддерживает создание пользователя и получение по email;
	Config - конфигурация сервиса, содержит URL подключения к БД, секрет JWT и срок жизни токена.

2.3 Проектирование межсервисного взаимодействия
Взаимодействие между сервисами организовано по двум различным схемам, каждая из которых соответствует собственному характеру сценария.
Синхронное взаимодействие через REST API используется для операций, где клиент ожидает немедленного ответа:
	deal-service проксирует запросы регистрации и входа в auth-service, что позволяет сохранить единую точку входа для клиентского приложения;
	при оформлении договора deal-service синхронно вызывает внутреннюю точку доступа payment-service POST /internal/invoices для выставления счёта; вызов защищён общим API-ключом, передаваемым в заголовке X-Internal-API-Key.
Асинхронное взаимодействие через брокер сообщений NATS используется для распространения доменных событий жизненного цикла:
	deal-service публикует события booking.confirmed и booking.cancelled при соответствующих операциях клиентского контура;
	payment-service публикует событие payment.completed при успешной обработке платежа;
	deal-service подписан на поток payment.completed и обновляет платёжный статус договора на paid.
	Каждое событие передаётся в формате конверта, содержащего идентификатор события, тип, время возникновения, источник, идентификатор трассировки и полезную нагрузку, что обеспечивает единообразную обработку событий на стороне потребителей. 
3	Реализация приложения
3.1	Реализация серверной части приложения
Серверная часть системы построена на основе микросервисной архитектуры, что обеспечивает масштабируемость, отказоустойчивость и возможность независимого развёртывания компонентов. В качестве основного языка программирования выбран язык Go, который обеспечивает высокую производительность и эффективное управление параллельными процессами, что критически важно для сервисов, работающих с внешним трафиком и событийными потоками.
Архитектурно система состоит из трёх взаимосвязанных микросервисов. Сервис аутентификации обеспечивает публичную регистрацию клиентов и выдачу JWT-токенов для последующей работы с защищёнными точками доступа. Сервис бронирований предоставляет каталог санаториев и полный жизненный цикл клиентских бронирований, а также служебный контур договоров, интегрированный с сервисом оплаты. Сервис оплаты отвечает за выставление счетов и идемпотентную обработку платежей.
В качестве HTTP-маршрутизатора используется Chi Router. Валидация JWT-токенов выполняется локально в каждом сервисе на основе общего секрета (HS256), что позволяет не обращаться к сервису аутентификации при каждом защищённом запросе. Все сервисы используют общий модуль platform-common, содержащий структурированное логирование на базе пакета slog, обёртку над NATS и middleware для метрик Prometheus и идентификаторов трассировки.
Вся инфраструктура контейнеризирована с использованием Docker и оркестрирована через Docker Compose. В состав стека входят контейнер PostgreSQL с автоматической инициализацией схем, контейнер NATS и контейнеры трёх микросервисов. Каждый сервис публикует свой HTTP-порт для внешнего доступа: auth-service - 8081, deal-service - 8082, payment-service - 8083.
3.2	Реализация клиентской части приложения
В реализованной версии системы доступны следующие сценарии.
Регистрация и вход клиента. Клиент отправляет запрос регистрации в deal-service, который проксирует его в auth-service. Сервис аутентификации валидирует входные данные (email, длину пароля не менее восьми символов, непустое имя), хеширует пароль алгоритмом bcrypt и сохраняет пользователя в таблице auth.users с ролью client. При последующем входе сервис аутентификации сравнивает хеш пароля и возвращает JWT-токен со сроком действия, заданным переменной окружения TOKEN_TTL_MINUTES.
Просмотр каталога санаториев. Запрос GET /api/sanatoriums в deal-service поддерживает фильтрацию по городу, медицинским профилям, максимальному расстоянию до моря, ценовому диапазону и датам проживания. При указании параметров check_in и check_out в выдачу попадают только санатории с доступной вместимостью на заданный период. Результаты возвращаются с пагинацией.
Создание и управление бронированиями. Клиент с действующим JWT-токеном (роль client) может создать бронирование, указав идентификатор санатория, даты и число гостей. Сервис бронирований выполняет двухуровневую проверку: на уровне приложения - корректность дат и положительного числа гостей, на уровне SQL-запроса - отсутствие превышения вместимости санатория с учётом пересекающихся подтверждённых бронирований. При успешном создании в NATS публикуется событие booking.confirmed. Операции изменения и отмены бронирования доступны только владельцу.
Оформление договора и выставление счёта. Сотрудник служебного контура (роль admin, manager или accountant) отправляет запрос на создание договора в deal-service. Сервис сохраняет договор в платёжном статусе invoice_requested, затем синхронно вызывает точку доступа POST /internal/invoices сервиса оплаты. При успешном ответе идентификатор созданного счёта сохраняется в поле invoice_id договора, а платёжный статус обновляется на invoice_issued. При ошибке вызова сервиса оплаты договор остаётся в статусе invoice_failed с описанием ошибки в поле payment_error.
Обработка платежа. Запрос на проведение платежа принимается по точке доступа POST /api/v1/payments. Сервис оплаты открывает транзакцию с уровнем изоляции Serializable, блокирует соответствующий счёт (SELECT FOR UPDATE), проверяет точное соответствие суммы платежа сумме счёта, сохраняет запись платежа и переводит счёт в статус paid. Повторный запрос с тем же ключом идемпотентности возвращает ранее созданный платёж без повторной обработки. После успешного завершения транзакции сервис оплаты публикует событие payment.completed в NATS.
Автоматическое обновление договора. Сервис бронирований подписан на поток payment.completed. При получении события он обновляет поле payment_status договора на paid. Таким образом, связь между договором и фактом оплаты устанавливается асинхронно и без блокирующего обратного вызова со стороны сервиса оплаты.
