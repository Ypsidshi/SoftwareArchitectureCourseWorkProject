4 Тестирование программного обеспечения

В рамках верификации разработанной информационной системы проведены функциональное тестирование методом чёрного ящика, нагрузочное тестирование и статический анализ метрик кода. Основной акцент сделан на серверную часть микросервиса бронирований и каталога (`deal-service`), реализующую критичный сценарий создания бронирования; сквозной пользовательский поток проверен под нагрузкой, сложность центрального метода сервисного слоя оценена количественными метриками.

4.1 Обоснование выбора видов тестирования и метрик кода

Тестирование программного обеспечения является неотъемлемой частью жизненного цикла разработки и направлено на проверку соответствия реализованного продукта заявленным функциональным и нефункциональным требованиям, а также на выявление дефектов до ввода системы в эксплуатацию.

Для распределённых приложений, построенных в стиле микросервисной архитектуры, применяются следующие виды тестирования.

Модульное тестирование проверяет отдельные функции и методы в изоляции от остальных компонентов. Интеграционное тестирование верифицирует взаимодействие модулей внутри сервиса и обмен данными между микросервисами. Функциональное тестирование подтверждает соответствие поведения системы спецификации с точки зрения внешнего наблюдателя (метод «чёрного ящика»). Нагрузочное тестирование оценивает время отклика и устойчивость приложения при имитации параллельной работы пользователей.

В рамках настоящей работы в качестве основного объекта детального анализа выбран метод `BookingService.CreateBooking` микросервиса `deal-service`, поскольку он концентрирует проверки обязательных идентификаторов, валидацию дат и числа гостей, обращение к репозиторию и публикацию события подтверждения бронирования. Сквозные сценарии с участием клиентского веб-приложения, сервиса аутентификации и сервиса оплаты отражены в нагрузочном тестировании (раздел 4.3, рисунок 23). Исходный текст метода `CreateBooking` приведён в приложении Е (листинг Е.1).

Для функциональной верификации применена техника построения функциональной диаграммы (cause-effect graph) и сокращённой таблицы решений. Для оценки эксплуатационных характеристик использован инструмент k6. Для количественной оценки сложности и сопровождаемости кода рассчитаны метрики объёма (SLOC), Холстеда, цикломатической сложности по Маккейбу, метрики Чепина и объектной метрики WMC; результаты автоматизированной проверки цикломатической сложности получены с помощью `gocyclo`.

4.3 Нагрузочное тестирование

В качестве тестируемого сценария выбран пользовательский поток, включающий регистрацию пользователя, вход в систему, получение списка санаториев и выполнение операции создания бронирования. В рамках одного тестового прохода последовательно выполнялись действия, характерные для практического использования приложения. Сценарий нагрузки реализован средствами k6; количественные результаты прогона отражены на рисунке 23.

Нагрузка в ходе тестирования изменялась поэтапно. На первом этапе в течение 30 секунд количество виртуальных пользователей увеличивалось до 10. Далее в течение 1 минуты нагрузка возрастала до 50 виртуальных пользователей, после чего в течение ещё 1 минуты поддерживалась на этом уровне. На завершающем этапе в течение 30 секунд нагрузка снижалась до 0.

В ходе тестирования заданы пороговые значения для ключевых запросов. Для запроса получения списка санаториев установлено требование: 99% запросов выполняются менее чем за 1 секунду. Для запроса создания бронирования — менее чем за 2 секунды. Для общего числа ошибочных HTTP-запросов задано ограничение не более 5%.

Метрика WMC (Weighted Methods per Class) определяет суммарную сложность всех методов класса. Для `BookingService` WMC = **40**, что отражает концентрацию бизнес-логики в одном типе. Наибольший вклад вносят `ListSanatoriums` (CC = 10) и группа методов создания, обновления и получения деталей санатория (CC = 6 каждый).

4.4.6 Автоматизированный подсчёт

Метрики цикломатической сложности верифицированы с помощью инструмента `gocyclo`. Фрагмент вывода команды для файла `booking.go`:

```text
6 service.(*BookingService).CreateBooking booking.go:163:1
...
Average: 3.85
```

Среднее значение цикломатической сложности по всем функциям файла: **3,85**.

Полученные показатели метрик кода находятся в пределах, приемлемых для сопровождения и тестирования: умеренный объём по Холстеду, цикломатическая сложность центрального метода CC = 6, метрика Чепина Q = 22, суммарная WMC = 40. Результаты функционального и нагрузочного тестирования, а также статического анализа подтверждают соответствие реализованного поведения заявленным требованиям к сценарию бронирования.
