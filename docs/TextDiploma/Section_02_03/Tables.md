Раздел 2. Проектирование

Таблица 1 – Сущности концептуальной модели данных

| Название сущности | Условный идентификатор | Назначение сущности |
|:------------------|:-----------------------|:--------------------|
| Пользователь | User | Учётная запись клиента или администратора системы |
| Санаторий | Sanatorium | Описание объекта размещения и лечения, параметры для фильтрации |
| Медицинский профиль | MedicalProfile | Справочник профилей лечения для персонализированного подбора |
| Бронирование | Booking | Заявка клиента на размещение в санатории на период дат |
| Счёт на оплату | Invoice | Документ на оплату, связанный с бронированием |
| Платёж | Payment | Факт оплаты счёта с ключом идемпотентности |
| Договор (служебный контур) | Contract | Резервная сущность служебного учёта размещения (при расширении B2B-сценариев) |

Таблица 2 – Связи между сущностями концептуальной модели

| Связь | Сущность 1 | Сущность 2 | Тип связи | Описание |
|:------|:-----------|:-----------|:----------|:---------|
| Пользователь – Бронирование | Пользователь | Бронирование | 1:N | Один клиент может иметь множество бронирований |
| Санаторий – Бронирование | Санаторий | Бронирование | 1:N | Один санаторий принимает множество бронирований на разные периоды |
| Санаторий – Медицинский профиль | Санаторий | Медицинский профиль | M:N | Санаторий специализируется на нескольких профилях лечения |
| Бронирование – Счёт | Бронирование | Счёт | 1:1 | Для клиентского сценария счёт выставляется на конкретное бронирование |
| Счёт – Платёж | Счёт | Платёж | 1:N | По счёту допускается учёт попыток оплаты; успешная фиксируется идемпотентно |
